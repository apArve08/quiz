P2 · WEEKS 5–8

Distributed Cache-Aside System

Redis · Cache invalidation · Event-driven · Race condition protection
⚡ Backend Node.js / Go ▼

What You're Building

  • PostgreSQL + Redis two-tier data layer in Docker
  • Cache-Aside middleware: Read Redis → miss → fetch PG → populate Redis
  • Atomic write layer: PUT/PATCH wraps SQL + Redis DEL in one transaction
  • k6 load test proving DB CPU drop with caching enabled

Core Concepts You'll Learn

  • Cache-Aside vs Write-Through vs Read-Through patterns
  • TTL design: too short = cache misses, too long = stale data
  • Dirty read problem and why atomic invalidation matters

Week-by-Week Breakdown

WK 05
Two-Tier Database Setup

Docker Compose with PostgreSQL + Redis side by side. Seed 20,000 records. Write basic CRUD endpoints in Node.js without caching first.

WK 06
Cache-Aside Middleware

Wrap GET endpoint: check Redis key first → if hit, return immediately → if miss, pull from PG and set entry with explicit TTL boundaries.

WK 07
Atomic Cache Invalidation

Build the write path. On PUT/PATCH: wrap SQL UPDATE in a transaction, on commit success fire Redis DEL for that key to preserve consistency blocks.

WK 08
Load Test & Proof

Write a k6 script tracking 3,000 virtual users ramping up over 60 seconds. Capture system resource graphs to isolate processing differentials.
